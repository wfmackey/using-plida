# using-plida

Quarto site demonstrating modern R data-engineering tools (arrow, duckdb, dbplyr, data.table) on larger-than-memory PLIDA-style administrative microdata. Hosted at https://wfmackey.github.io/using-plida/.

## Data locations

No big data lives in this repo. All intermediate data is at `/Users/willmackey/offline/using-plida-data/`:

- `parquet/` — arrow-streamed parquet files from fplida CSVs
- `duckdb/plida_tables.duckdb` — main materialised DuckDB database (68 GB, includes 3.5B STP rows)
- `duckdb/plida_views.duckdb` — views-over-parquet variant
- `duckdb/plida_variants.duckdb` — storage layout experiment variants (45 GB)
- `duckdb_tmp/` — DuckDB temp spill directory (clean up after runs)

Source CSVs are at `/Users/willmackey/offline/fplida-data/fplida_30m_csv/` (751 GB, generated by `fplida`).

## R scripts (in dependency order)

1. `R/00-paths.R` — centralised paths, never change these without updating everything
2. `R/helpers.R` — timing (`rss_gb`, `time_it`, `log_result`)
3. `R/01-convert-csv-to-parquet.R` — converts ITR + MBS CSVs to parquet
4. `R/01b-convert-supplementary.R` — converts demo + spine CSVs
5. `R/06-convert-stp.R` — converts STP FY2021 + FY2122 + jobs + spine
6. `R/06b-convert-stp-remaining.R` — converts STP FY2223, FY2324, FY2425
7. `R/02-build-duckdb.R` — builds plida_views.duckdb and plida_tables.duckdb
8. `R/07-build-duckdb-stp.R` — adds STP tables to plida_tables.duckdb (year-by-year sorted ingest)
9. `R/03-benchmarks.R` — benchmark functions per (task, backend)
10. `R/04-run-benchmarks.R` — driver that runs each pair in a fresh subprocess
11. `R/05-storage-variants.R` — builds plida_variants.duckdb with sorted/intkey/PK variants
12. `R/05b-run-variant-tasks.R` — runs point/narrow/full/intkey tasks on variants
13. `R/08-benchmark-window-vs-join.R` — LAG vs self-join vs chunked benchmarks

## Key findings baked into the site

- **CSV → parquet:** 44.5 GB in 122s, peak RSS 1.76 GB (arrow streaming)
- **Groupby 170M rows:** dplyr 35s/14.5 GB vs DuckDB 0.16s/0.25 GB (200×)
- **3-way join 372M rows:** only DuckDB completes (3.5s/2.3 GB); dplyr, data.table, and arrow all OOM
- **Sorted table point lookup:** 47× faster than unsorted
- **Integer hash join key:** 2× faster than string key
- **Window functions on 700M rows:** LAG and self-join both OOM. Chunking into 10 person-hash-buckets (~70M each) works: ~2s per bucket
- **Views over parquet vs materialised tables for one-off pipelines:** views are 3× faster total (skip 170s materialisation)

## Things to watch out for

- `_system/base-spine.csv` in fplida can disappear during regeneration. The project uses `abs-core/plidage-core-demog` as the demographic spine instead.
- `ato-pit_itr/` contains 4 distinct schemas (context, ded-exp-off, inc-loss, whld-debt) that must be split into separate parquet subdirs before DuckDB ingest.
- DuckDB `memory_limit` should always be set explicitly (~20 GB on a 32 GB machine). Default will claim all RAM and collide with R.
- Synthetic data (`fplida`) does not model job switching — all STP switch counts are 0. The pipeline and timings are real; the results are not.
- CS04 creates and deletes a temporary `cs04_working.duckdb` during render. Don't interrupt the render mid-flight.
