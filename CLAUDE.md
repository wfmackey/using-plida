# using-plida

Quarto site demonstrating modern R data-engineering tools (duckdb, dbplyr, arrow, data.table) on larger-than-memory PLIDA-style administrative microdata. Hosted at https://wfmackey.github.io/using-plida/.

## Audience and tone

**Audience:** R tidyverse programmers — comfortable with `dplyr`, `ggplot2`, `purrr`, tibbles, piping — who have limited exposure to database internals, CS jargon, or SQL.

**Tone:** Plain, direct, explanatory. Every chapter should make sense to someone who has never opened a DuckDB file or thought about parquet row-groups. Case study pages use a conversational voice modelled on [R at Grattan](https://grattan.github.io/R_at_Grattan/). Reference and benchmark pages are slightly more formal but follow the same plain-language principle.

**Rules when writing for this project:**

- Tidyverse R is assumed. Jargon from CS / database internals is not.
- When a technical term is unavoidable, explain it in plain language and say why it matters. Examples of terms that need explaining: row group, zonemap, hash join, materialise, lazy, compute, chunk, partition, view, spill-to-disk, parquet, dbplyr translation.
- Prefer concrete descriptions over formal ones ("DuckDB notices it's running out of memory and writes pieces to a scratch folder on disk" beats "DuckDB spills hash-join intermediate state to temp storage when it exceeds `memory_limit`").
- Avoid big-O notation. Describe the scaling in rows and seconds and GB of RAM.
- Do not use `OOM` in prose — write "out of memory" or "the process is killed".
- Percentages are written without a space before `%` (e.g. `60%`, not `60 %`).

## Code style

**Tidyverse first. Always.**

- Every analytical query in the main chapters and case studies is written in `dplyr` verbs against a `dbplyr` lazy handle (`tbl(con, "name") |> rename_with(tolower)`).
- Window functions use `group_by() |> window_order() |> mutate(lag(...))` and rolling frames use `window_frame(from, to)`.
- Joins are `inner_join`, `left_join`, `semi_join`, `anti_join`, `full_join`. `semi_join` replaces `filter(x %in% ...)` in every case involving large data.
- `compute()` materialises a lazy pipeline to a DuckDB table without leaving the tidyverse. Use it before reusing an intermediate (e.g. each bucket in the chunked LAG pattern).
- Date arithmetic uses `lubridate` (`floor_date()`, `days(n)`) — these translate to DuckDB cleanly.

**SQL appears only in unavoidable one-off setup:**

- Connection pragmas: `SET memory_limit = '20GB'`, `SET threads TO 8`.
- CSV → parquet conversion: `COPY (SELECT * FROM read_csv_auto(...)) TO 'path' (FORMAT PARQUET, ...)`.
- `CREATE OR REPLACE VIEW` / `CREATE OR REPLACE TABLE` to register or materialise a parquet glob as a DuckDB table. Wrap in a helper (`register_view()`).

**Anti-patterns to avoid:**

- `dbGetQuery(con, "SELECT ...")` with a hand-written multi-line SQL query for analysis. Almost never the right answer.
- `sql("...")` fragments inside a dplyr pipeline for anything that has a tidy equivalent. Reserved for genuine DuckDB-specific features (e.g. `ASOF JOIN`). When doing this, explain what you're doing and why it doesn't have a tidyverse equivalent, referring to the dbplyr (or equivalent) docs.
- `filter(x %in% (y |> pull(key)))` on large right-hand sides. This pulls the key column into R memory. Use `semi_join()`.
- `collect()` mid-pipeline on anything large. Keep the pipe lazy until the last, small step.
- Expressing `hash(x) %% n == b` through dbplyr on large data. Pre-compute the bucket column in the VIEW definition once.
- Carrying all columns through a window function. Use an early `select()` to narrow to the columns the window and downstream filters need.

## Key findings baked into the site

- **CSV → parquet (DuckDB):** 44.5 GB in ~65s, peak memory under 3 GB. About 2× faster than Arrow on the six-file benchmark, with slightly smaller output parquet.
- **Groupby on 170M rows:** dplyr-on-tibbles ~35s / 14.5 GB vs DuckDB ~0.16s / 0.25 GB (200×).
- **Three-way join on 372M rows:** only DuckDB completes (~3.5s / 2.3 GB). dplyr, data.table and Arrow all run out of memory.
- **Sorted fact table, single-person lookup:** 47× faster than unsorted.
- **Integer join key:** about 1.9× faster than a 12-char string key.
- **Window functions on 700M rows:** `lag()` and self-join both run out of memory. Chunking into 10 person-hash buckets (~70M each) works: ~2s per bucket, ~20s total.
- **Views over parquet vs materialised tables for a one-off pipeline:** views are ~3× faster end-to-end (the 170s ingest dominates). Materialise for repeated workloads.


User needs to `gh auth login` before first deploy. Site URL: https://wfmackey.github.io/using-plida/.

When Quarto `freeze: auto` is on, re-render only the files you changed by clearing `_freeze/<name>/` for each and running `quarto render <file>.qmd` individually. **Do not** pass multiple `.qmd` files to a single `quarto render` invocation — Quarto will concatenate them into a single pandoc call and produce one corrupt output HTML.

## Things to watch out for

- `_system/base-spine.csv` in fplida can disappear during regeneration. The project uses `abs-core/plidage-core-demog` as the demographic spine instead.
- `ato-pit_itr/` contains 4 distinct schemas (context, inc, ded, whld) that must be split into separate parquet subdirs before DuckDB ingest. In the DuckDB database they are tables `itr`, `itr_inc`, `itr_ded`, `itr_whld`.
- DuckDB `memory_limit` must always be set explicitly (~20 GB on a 32 GB machine). Default will claim all RAM and collide with R.
- Synthetic data (`fplida`) does not model job switching — all STP switch counts are 0. The pipeline and timings are real; the results are not.
- CS04 creates and deletes a temporary `cs04_working.duckdb` during render. Don't interrupt the render mid-flight.
- In cs04 and in the chunked LAG pattern generally: pre-compute the `bucket` column in the VIEW definition (as an integer `CAST(hash(x) % n AS INTEGER)`). Doing the hash/modulus inside the dbplyr pipeline on every iteration defeats chunk-skipping and OOMs on 70M-row windows.
